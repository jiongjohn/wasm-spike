## 1. Visual Identity Statement

### 一句话视觉规则

> **FC 红白机里的塔:一切素材锁在 16×16 像素格与 32 色复古盘之内——唯独成长反馈不受约束,必须刺穿那层怀旧滤镜。**

**用法**:任何视觉决策有歧义时,先问「它符合 FC 时代的低保真约束吗?」然后问「如果它是成长反馈,它有没有刺穿背景?」两问都答得了,决策就有了。

---

### 视觉身份声明

《像素魔塔·无尽塔》是一座被红白机像素困住的塔——克制的石墙、方块的怪物、有限的颜色,一切都忠于三十年前的 FC 内存约束。

但每一次变强,这座塔就会短暂地背叛自己:金色飘字刺穿灰墙,升级闪光撕开暗调,属性数字跳涨的那一帧比所有背景都更真实。

这就是它的视觉身份:**怀旧是外壳,爽感是核心**——克制与爆发的张力永远共存,从不消解。

---

### 3 条支撑原则

#### 原则 A — 「格即法」(Grid Is Law)
所有视觉素材——角色、怪物、道具、墙体、UI 图标——必须能在 16×16 像素单元格中读清楚,且在网格坐标系上精确对齐。

*设计测试*:当有人想给某道具加一个半格宽的装饰光晕时,这条原则说:光晕要么对齐格边,要么删掉。任何"感觉更好看"的偏移,若打破网格对齐就砍掉。

*服务支柱*:**P3「三秒上手」**。网格天然生产视觉秩序——玩家不用思考"那个东西在哪",它一定在某个格上,扫一眼就读懂场地。

#### 原则 B — 「调色板封印」(Palette Prison)
全局只允许一个 ≤32 色的复古调色板;向调色板添加新色须先检索复用,只有在功能语义无可替代时才允许替换使用率最低的旧色。

*设计测试*:当想为新 Boss 增加一个"更酷的渐变橙"时,这条原则说:先检索调色板现有橙/黄系色,能否组合出足够区分度?若能,禁止新增;若不能,替换一格最低频旧色。

*服务支柱*:**P4「每层都有新发现」(产生张力)**——每层要有新元素意味着有引入新视觉语言的冲动;调色板封印迫使新发现必须在既有符号系统内表达(靠形状变化+动画节奏区分新怪),倒逼更有创意的像素造型,而非靠堆色解决。

#### 原则 C — 「反馈破界」(Feedback Breaks the Rules)
伤害飘字、升级闪光、属性跳涨等成长反馈被明确豁免于「克制复古」的整体调性——必须在亮度、饱和度、动态上超出环境,哪怕制造出风格断层。

*设计测试*:当有人说「升级闪光太亮,和整体怀旧风格不搭」时,这条原则说:搭不搭不是标准,「玩家在打怪的瞬间有没有立刻感知到变强」才是。闪光不够亮就变亮,直到在 5 英寸低端机小屏幕上不用找就能看到它。

*服务支柱*:**P1「看得见的成长」(真北)**。这条原则是真北在视觉层的物理实现;它与 A、B 形成刻意的张力——静态素材受双重约束,成长反馈明确解除那些约束。这个张力就是「怀旧外壳包裹爽感核心」这一游戏身份在视觉上的直接映射。

---

### 三原则张力裁决表

| 冲突场景 | 裁决 |
|---|---|
| 新楼层要有新怪,想加调色板外的新颜色 | B 胜:用形状/像素剪影区分,不加色 |
| 升级闪光需要高饱和亮色,感觉违反复古调性 | C 胜:成长反馈豁免,仅限动态反馈素材,不影响静态环境 |
| 某道具在 16×16 格内轮廓读不清,想扩大到 24×24 | A 胜:优化像素剪影,必要时整体提格(须统一,不可单个破格) |

## 2. Mood & Atmosphere

> 所有实现手段限定在 FC 像素约束内:调色板色替换、帧动画节奏、静帧/动帧对比度差异。
> 不依赖任何引擎光照特效。

### 六状态氛围定义

#### 状态 1 — 探索层(Exploration)
**情绪目标**:规划性好奇——驱动力是「我想看看下一格是什么」,不是焦虑,是主动扫描、蓄力决策的从容。

| 维度 | 定义 |
|---|---|
| 描述词 | 沉稳 / 克制 / 可读 / 蓄力 / 规整 |
| 整体亮度/对比 | 中低亮度,石墙地板用调色板暗色段;可交互元素比环境亮 1–2 阶,靠色差不靠发光 |
| 角色可读性 | 玩家角色用调色板内最高对比度静态色组,与地板/墙体拉开差距,任意位置一眼定位 |
| 动画节奏 | 2–4 fps 的环境微呼吸(火把等),角色待机小幅抖动;画面以「棋盘等待落子」的静止感为主 |
| 状态信号 | **画面最静**——这是基准态,其他所有状态都从这里偏离 |

#### 状态 2 — 战斗中(In Combat)
**情绪目标**:紧绷的算盘——紧张来自「数字对不对」而非「运气如何」,是智识上的专注。

| 维度 | 定义 |
|---|---|
| 描述词 | 紧凑 / 节律 / 锋利 / 交替 / 可控 |
| 亮度变化 | 背景压暗 1 档或简化细节,让角色与敌人的对撞关系成为焦点 |
| 撞击帧 | 撞击瞬间角色/敌人各自高亮 1 帧(调色板最亮色替换轮廓),随即恢复——模拟撞击震 |
| 数值区域 | 血条/伤害预演保持全屏第二高对比度(第一是成长反馈) |
| 动画节奏 | 每回合 0.3s 一次攻击帧序列,攻击–受击–结算固定节拍,类 16 分音符打击感 |
| 状态信号 | 背景轻微压暗 + 角色/敌人面对面布局 + 节律帧序列启动;不靠弹窗,靠布局收缩 |

#### 状态 3 — 成长反馈(Growth Feedback)
**情绪目标**:「我变强了这一刻必须被看见」——爽感的物理化,视觉无可回避。

| 维度 | 定义 |
|---|---|
| 描述词 | 爆发 / 穿透 / 金亮 / 瞬时 / 满足 |
| 亮度 | **全游戏最高** — 明确违反探索/战斗的低调整体调性(Principle C 豁免) |
| 升级闪光 | 全屏闪白 1–2 帧后恢复——**全游戏唯一允许「全屏提亮」的时机** |
| 伤害飘字 | 比 UI 常规文字大 1.5–2 倍,金/黄色,带 1px 暗描边;在任意背景可读 |
| 属性跳涨 | 绿色上浮飘字,速度比普通飘字快 30%——视觉上是「蹦出来的」而非「滑上去的」 |
| 持续时间 | 峰值 ≤0.5s,之后迅速回落到探索/战斗基准;短促才有冲击力 |
| 状态信号 | 单帧级亮度峰值 + 突然动态爆发;玩家不用读文字就知道「发生了好事」 |

#### 状态 4 — 卡关/死亡(Stuck / Dead)
**情绪目标**:「这局打完了,我知道为什么」——死亡是可分析的结论,玩家离开时清醒而非沮丧。

| 维度 | 定义 |
|---|---|
| 描述词 | 暗沉 / 静止 / 冷却 / 收缩 / 清晰 |
| 亮度 | **全游戏最低** — 全屏压到调色板最暗 4–5 色段,像画面「熄灯」 |
| 角色 | 替换为低饱和度灰化版本,与环境对比度降低——主角「消失在背景里」 |
| 结算 UI | 死亡结算卡反而是此状态最高对比度区域——把注意力引向「复盘」而非「悲伤」 |
| 动画节奏 | 所有动画停止;保持 0.5s 静帧缓冲,再淡入结算 UI——刻意停顿,不是 bug |
| 状态信号 | 动画全停 + 亮度触底;探索是中亮微动,死亡是最暗全静 |

#### 状态 5 — 新层到达(New Floor)
**情绪目标**:「新的还是可期的」——上楼前吸一口气、新环境扑面而来的兴奋期待。

| 维度 | 定义 |
|---|---|
| 描述词 | 清新 / 转场 / 微亮 / 短暂 / 开幕 |
| 转场动画 | 全屏从暗到亮的 2–3 帧淡入序列(调色板最暗色→正常亮度),像幕布拉开 |
| 初见高光 | 新楼层第一帧比探索基准亮 0.5 档,持续约 0.8s 后自然衰减 |
| 新元素提示 | 当层首次出现新怪物/新地形时,该元素短暂闪烁 2 帧——「初见提示」 |
| 楼层编号 | 楼层数 UI 短暂浮现并淡出——游戏内少数主动告知信息的时机 |
| 动画节奏 | 0.5–1s 亮度/动画小高潮,随即平滑衔接探索态;不能太长否则打断爬塔心流 |
| 状态信号 | 方向性推移/淡入动画——探索层内移动时不存在这种方向性 |

#### 状态 6 — 胜利(Victory)
**情绪目标**:「我真的打通了」——FC 游戏通关时的那种成就感,带着回望整座塔的自豪。

| 维度 | 定义 |
|---|---|
| 描述词 | 辉煌 / 克制 / 温暖 / 沉淀 / 值得 |
| 色调转变 | 调色板内暖色(金/橙)替换全局石墙灰蓝调——**全游戏唯一暖色调画面**,纯靠调色板色替换 |
| Boss 击倒瞬间 | 触发「成长反馈」级别亮度峰值——最终击倒是最大成长反馈 |
| 胜利画面亮度 | 比探索态亮 1 档,维持全屏 |
| 主角展示 | 32×32 大号静态 sprite + 像素化 8 方向放射光芒线(帧动画旋转)——FC 奖励画面经典语言 |
| 动画节奏 | 辉煌 0.5s 后降速到比探索更慢的节奏;快慢对比让「到达终点」有重量感 |
| 状态信号 | 暖色调切换是最强状态信号——不用读文字,色调变化本身宣告「通关了」 |

---

### 六状态对比速查

| 状态 | 亮度 | 对比度 | 动画节奏 | 色调 | 核心信号 |
|---|---|---|---|---|---|
| 探索层 | 中低 | 中 | 慢(微呼吸) | 冷灰蓝 | 画面最静 |
| 战斗中 | 中低 | 中高 | 节律爆发 | 冷灰蓝+角色高光 | 节律帧序列 |
| **成长反馈** | **峰值** | **最高** | 瞬时≤0.5s | **金/亮黄** | 唯一全屏亮度冲击 |
| 卡关/死亡 | **最低** | 低→高(结算) | 完全静止 | 灰化 | 动画全停 |
| 新层到达 | 中低+微高 | 中 | 短促上扬 | 冷灰蓝+转场 | 方向性淡入 |
| 胜利 | 中高 | 高 | 辉煌→沉淀 | **暖金/橙** | 唯一暖色调 |

## 3. Shape Language

### 3.1 角色/怪物轮廓哲学

**核心规则:剪影先于颜色,格内读清楚,缩略图下不混淆。**

**剪影识别测试(必须通过)**:把贴图填成纯黑放在 16×16 格里,从 5cm 距离看——0.5 秒内能识别出「这是什么类型」。通不过就重新造型,不靠调色板补救。

**四类型剪影区分特征**

| 类型 | 首要剪影特征 | 尺寸 | 禁止 |
|---|---|---|---|
| 玩家角色 | 正立人形:宽肩、头占格高 40%、倒梯形外轮廓 | 13–14px 高 | 弯腰/侧身/不规则外轮廓 |
| 杂兵怪物 | 低矮横宽:高度 ≤10px,宽度 >12px,横向压扁感 | 8–10px 高 | 高度接近玩家 |
| Boss | **充格或双格**:占 32×32(2×2 格);外轮廓有 2+ 明显突起(角/翼/肢) | 28–32px 高 | 矮平轮廓 |
| 道具/钥匙 | 几何形+空心感:圆/菱/矩形为主,中心留空白或高亮像素 | 8–12px | 有机曲线/生物感轮廓 |

**高度阶梯原则**:玩家 > Boss(凭体积) > 杂兵——高度绝对不能倒置。这是原则 A「格即法」在角色层级上的物理映射。

**轮廓锯齿语言**:FC 像素角色的锯齿边缘是风格标记,阶梯数控制感知:
- 玩家:3–4 阶清晰锯齿 → 有力/明确
- 杂兵:1–2 阶细碎锯齿 → 细小/快速感
- Boss:4–6 阶粗重锯齿 → 沉重/威压感

---

### 3.2 环境几何语言

**核心规则:直角主导,圆弧用于信号,格边是物理法律。**

所有静态环境元素以**直角与 45° 斜切像素**为绝对主导;不允许超过 2px 的弧线出现在环境轮廓上。

**可走格 vs 不可走格:视觉区分三段法**

| 类别 | 亮度段 | 像素密度 | 关键词 |
|---|---|---|---|
| 地面/可走格 | 调色板中段(中灰蓝) | 低(≤4px 变化/格) | 空、平、可落 |
| 墙体/不可走格 | 调色板暗段(深灰蓝) | 高(8–12px 变化/格) | 实、密、压 |
| 门/特殊格 | 调色板中高段(比地面亮 1–2 阶) | 中(有 2px 描边) | 亮、有边、可交互 |

暗且密 = 走不了;中亮且空 = 可走;有描边且更亮 = 可触发——玩家无需读图例即可感知。

**视觉层级:三层深度**

- **后景/墙体**:最暗 + 最高细节密度 → 视觉重量压住画面底部
- **地面**:中灰 + 低细节 → 中性通道,不抢注意力
- **前景道具/角色**:最高对比度 + 1px 描边 → 「浮在」地面上

**门的几何语言(唯一例外)**:门允许用 2 个 45° 斜切像素模拟门洞顶部轮廓(≤4px 斜切面)。「直角=墙,有斜切=出口」——无需文字提示识别门框。

---

### 3.3 UI 形状语法

**核心规则:UI 与游戏世界同调——像素化、直角、格对齐。**

UI 面板/按钮不使用独立 HUD 风格(圆角磨砂/现代扁平/渐变背景),与游戏世界形状语言完全统一。手机小屏下分裂的视觉风格会消耗认知带宽,违反 P3「三秒上手」。

**边角处理**:全直角为基准。唯一例外:**可点击按钮**四角允许 1px 单像素缺角(伪像素倒角)作为「可交互」的一致性识别标志——FC 游戏菜单的经典语言。

**图标 / 数字 / 文字搭配**

| 元素 | 尺寸 | 位置 | 优先级 |
|---|---|---|---|
| 图标(道具/属性) | 8×8 或 16×16px | 左对齐/顶部居中 | 次要(识别用) |
| 属性数字 | 等宽像素字体,最小 8px 高 | 紧靠图标右侧 1–2px | **主要** |
| 文字标签 | — | 图标+数字下方或侧边 | 辅助 |

数字永远比图标对比度更高——看数字变化是核心循环(P1),数字是第一等公民。

**触控热区**:抖音小游戏触控平台,所有可点击 UI 元素物理触控区域 ≥ 44×44px;视觉像素可以更小,触控热区比视觉图形大,周围留透明缓冲——解决「像素按钮视觉小但触控准」的矛盾。

**字体**:等宽像素字体,具体字型在 Section 7 UI/HUD 节锁定。

---

### 3.4 视觉焦点层级

**固定层级:成长反馈飘字 > 玩家角色 > 可交互道具/门 > 地面 > 墙体。任何情况下不变。**

**英雄形状(Hero Shapes)**——眼睛最先落点的三类:
1. **玩家角色当前格**:调色板内最高对比度静态色组 + 高亮色+暗色轮廓双层描边
2. **成长反馈飘字**:Z 层最高渲染,任意背景下 1px 暗描边保证可读
3. **下一层出口/楼梯**:与地面同格但加 2px 高亮外框描边——「前进方向」的长期视觉锚

**支撑形状(Supporting Shapes)**——退到后景的三类:
1. 地面空格:中灰蓝/低细节/无描边——画布而非内容
2. 已消灭/已拾取格:空格处理;绝不留残影或染色标记
3. 墙体纹理细节:暗色段砖缝/裂纹——质感存在但宁缺勿抢前景

**成长反馈的形状语言(Principle C 的形状层实现)**

| 反馈类型 | 形状特征 | 理由 |
|---|---|---|
| 伤害飘字 | 加粗等宽像素字,大 1.5–2 倍,向右上 45° 飘移 | 45° 方向在横竖直角世界里是「异质的」——方向本身制造冲击 |
| 升级闪光 | **全屏白色矩形帧**,1–2 帧全覆盖后消失 | 矩形是最简单的 FC「屏幕爆炸」;与塔的外框同构——整个世界在响应成长 |
| 属性跳涨 | 小字块垂直上移后消失,带 1px 暗描边 | 垂直=「上升」直觉映射;形状与 HUD 数字对应,强化「那个数变大了」的因果链 |

**Gestalt 支撑**:飘字浮在角色格正上方(接近律——出现位置与触发位置接近,因果关系无需教学);所有可交互元素统一用「亮色+描边」标记(相似律——有描边的都能碰)。

## 4. Color System

> 采用**自定义 FC 风格 32 色固定盘**,非直接套用 NES 硬件 64 色盘。
> 每格色槽都服务于「背景克制 + 反馈突破」的双极张力。

### 4.1 主调色板结构

调色板分三段,共 23 格实际使用,保留 9 格缓冲供后期主题扩展。

#### 暗色段(Dark Band)— 5 色,环境基底

| 色槽 | 描述 | 主要用途 |
|---|---|---|

### 4.2 语义色汇表

#### 角色与敌对单位
- **玩家角色 → 暖肤/米橙(M7)**:冷灰蓝的塔身底色下,玩家是唯一暖色系角色。Gestalt 异质突出——冷环境里的暖色立刻被识别为「主体」。**形状备份**:明确人形剪影,色盲也能识别。
- **普通怪物 → 饱和红(M5/M6)**:红色=危险是最原始的视觉信号(血液/警告)。魔塔玩家已内化「红色=敌对」。**形状备份**:非人形剪影(兽形/骷髅形)。
- **Boss → M5/M6 为主,M9 橙红为副**:与普通怪同属危险语义共享红色系;区分靠**尺寸**(32×32 占 4 格),不消耗新色槽。

#### 钥匙与门
- **黄钥匙/黄门 → M1 黄金 + M2 亮黄高光**:黄金=解锁/奖励/财富是文化通义;经典魔塔已建立玩家认知,无需重建。**形状备份**:钥匙=齿形轮廓,门=门框矩形。
- **蓝钥匙/蓝门 → M3 天蓝 + M4 冰蓝高光**:蓝=魔法/高级通道,与黄形成「普通/特殊」等级差。黄蓝对比在红绿色盲下安全(走蓝-黄视觉通道)。**形状备份**:蓝门用「弧形/圆顶」轮廓区分黄门的「矩形平顶」。

#### 道具与拾取物
- **HP 药水 → 中性绿(M8)**:绿=生命/恢复是游戏行业通用语义;在魔塔功能色体系中几乎唯一空缺,无歧义占用。**形状备份**:圆瓶剪影。
- **ATK 装备 → 橙红(M9)**:介于攻击性红色和奖励性黄色之间,传达「力量增益的奖励」。**形状备份**:剑/刺穿形状。
- **DEF 装备 → 冷灰蓝(M10)**:蓝灰=坚硬/金属/护甲的材质感;与攻击橙冷暖对立强化「攻/守」认知区分。**形状备份**:盾牌/方块形状。
- **钥匙碎片(软通货) → 亮橙(B3)+ 黄金(M1)**:碎片是黄钥匙的衍生物,沿用黄金色系;B3 亮橙用于碎片飘字,视觉暗示「不完整但有价值」。**形状备份**:不规则多边形碎块。

#### 成长反馈飘字(三属性三色)

| 飘字类型 | 颜色 | 备注 |
|---|---|---|
| ATK+ | B2 高饱和亮黄 | 攻击=最突出的成长感,用最亮黄 |
| DEF+ | B5 亮冰蓝 | 防御=护盾感,蓝色 |
| HP+ | M8 绿亮化版 | 回血=生命绿 |
| 受到伤害(普通) | B1 白 | 最高可读性 |
| 受到伤害(重伤≥30%HP) | B4 亮红 | 危险/急迫 |
| 碎片获得 | B3 亮橙 | 软通货区分于成长反馈 |

### 4.3 色盲安全检查

| 冲突对 | 风险 | 必须实施的备份 |
|---|---|---|
| 红怪(M5) vs 绿药水(M8) | **高**(红绿色盲,~8% 男性) | **形状必须不同**:怪物=非人形剪影,药水=圆瓶形 |
| ATK 橙(M9) vs HP 绿(M8) | 中(红绿轴) | ATK=剑形,HP=圆瓶形,形状差异足够大 |
| 升级亮黄飘字(B2) vs 金钥匙(M1) | 低-中(蓝黄色盲) | 飘字有动画(上浮消隐),钥匙为静态图标;时间维度区分 |
| 蓝门(M3) vs DEF 装备(M10) | 低 | M3 高饱和亮蓝 vs M10 低饱和冷灰蓝;形状强制区分:门=门框形,装备=盾牌形 |

### 4.4 UI 调色板 vs 世界调色板

- UI 主要使用亮色段(B1–B8)
- 世界主要使用暗色段(D1–D5)和中间段(M1–M10)

**硬规则**:UI 层所有元素明度必须高于游戏世界环境(D1–D5 段)——B6 深蓝黑背景 + B1 白文字保证 UI 永远不会「沉入」背景。

| DEF 数值 | M10 冷灰蓝(颜色=语义) |

### 4.5 调色板扩展规则(Principle B 操作协议)

**可触发调色板扩展的条件**:
- 新地形主题(Pillar 4,如熔岩/冰雪层) → 申请 1–2 格缓冲槽
- 新怪物语义无法复用现有色 → 替换 1 格最低频旧色
- 色盲可用性测试发现严重混淆 → 替换冲突色

**禁止触发**:「这个颜色更好看」/「想要更丰富效果」/MVP+VS 阶段(此阶段调色板**冻结**)

**🔒 锁定色(不可被顶替)**:D1 / M1 / M3 / M5 / M7 / M8 / B1 / B2 / B4 / B6 / B8——共 11 格。

替换操作须经 Art Director 审批:确认不引入语义歧义或色盲新冲突,并更新本节色表。

## 5. Character Design Direction

### 5.1 玩家角色视觉原型

**类型:通用冒险者(勇者原型)** — 不指定具体职业,保持剪影通用性,玩家自我投射。外观锁定为「FC 勇者」标准:头部占格高 40%、斗篷/披风宽肩拉宽倒梯形、右手持小剑/法杖(1–2px 剑身)。

**主角感靠三点独占性实现(不靠多色)**

1. **颜色独占**:M7 暖肤色是全局唯一暖色调角色,冷灰蓝背景下自动跳出
2. **高度独占**:13–14px 是全局唯一高度段,任何时候绝不与怪物(8–10px)混淆
3. **描边独占**:玩家角色用双层描边(外层 D1 深炭灰 + 内层 M7 暖色过渡);怪物只用单层轮廓描边

**推荐配色(全部在 32 色内)**

| 身体部位 | 主色 | 高光/阴影 |
|---|---|---|
| 肌肤/头部 | M7 暖肤 | M7 亮一阶 |
| 盔甲/服装 | M10 冷灰蓝 | D3 中深灰(阴影) |
| 武器 | M9 橙红(剑)或 M3 天蓝(法杖) | B1 白(高光刃) |
| 斗篷/头巾 | D2 深冷灰紫 | D3 边缘 |

**禁止**:玩家角色使用 M5/M6 红色(危险语义归怪物);使用 B2/B4 亮色段(成长反馈豁免色,不得静态使用)。

---

### 5.2 怪物视觉设计规则

**普通怪物剪影多样性(10px 高度内)**

分化靠 4 个独立维度,每种新怪物至少占据 2 个不同维度:

| 维度 | 变量 | 示例 |
|---|---|---|
| 顶部轮廓 | 平头/尖角/圆顶/多刺 | 骷髅=平头圆弧;史莱姆=圆顶;蝙蝠=双翼V形 |
| 宽高比 | 扁宽(12×8)/方(10×10)/窄高(8×10) | 同高度宽度可差 4px 以上 |
| 肢体附属 | 无肢/单侧突出/双侧对称/底部多足 | 鬼魂=无肢飘逸边缘;骑士=左盾右剑双侧 |
| 内部填充率 | 实心/中空/有洞 | 史莱姆=中空圆圈;骷髅=眼洞空格 |

**操作规则**:新怪物先画纯黑剪影,对照已有怪物——0.5 秒无法区分则重画,不进入配色阶段。

**怪物配色**:主色锁定 M5+M6 红色系。内部装饰允许借用 M10 冷灰蓝(金属感)或 D3(骨骼感)作点缀(面积 ≤30%)。**禁止**使用 M7(玩家独占)、M1/M3(钥匙道具独占)、B2/B4(成长反馈独占)。

**Boss(32×32)压迫感实现规则**

1. **填格率 ≥75%**:外轮廓覆盖 32×32 格 75% 以上像素面积
2. **3+ 突起**:外轮廓必须有至少 3 个明确突起(角/冠/翼/爪)
3. **垂直主轴**:头部/核心位于 32×32 上半部分 1/3 处,重心压低
4. **内部密度差**:Boss 内色彩变化节点比普通怪多 2 倍,仍限于 M5/M6/M9 红橙系
5. **Boss 专属**:允许 2×2 像素对称内部图案(眼珠/印记);受击时全身替换 B1 白色 1 帧(普通怪只做轮廓高光)

**新楼层「视觉新发现」的来源(不靠加色)**

按优先级:① 剪影原型替换(最强) → ② 内部像素分布节奏差异 → ③ 待机动画节奏差异(2fps 重型 vs 4fps 轻型) → ④ 借用道具功能色作怪物装饰(如持蓝色法球的魔法师)

---

### 5.3 表情/动作风格

角色情感**不靠面部**(16×16 格内面部区域极小),靠**身体姿势帧+动画节奏**。

| 情感状态 | 表达手段 | 帧数 |
|---|---|---|
| 待机 | 上下微移 1–2px,武器轻微抖动 | 2 帧循环 |
| 移动 | 底部腿轮廓左右交替 | 2 帧循环 |
| 攻击 | 整体前移 2px + 武器前刺(伸长 2–3px) | 2 帧(攻击+恢复) |
| 受击 | 整体后移 2px + 全身替换 B1 白色 1 帧(受击闪白) | 2 帧(闪白+恢复) |
| 死亡 | 替换为灰化版本(D3/D4 替换 M7/M10)→ 下移 4px 消失 | 3 帧 |
| 升级 | 整体上移 2px;升级闪光由单独全屏帧承担(非角色 sprite) | 1 帧 |

**硬规则**:角色 sprite **最多 3 帧**;复杂反馈情绪由独立特效层承担。普通怪 2 帧(待机循环);Boss 4 帧(待机 2 + 攻击 1 + 受击 1)。

---

### 5.4 LOD / 缩放考量

16×16 素材在主流手机上以约 **2.5x 倍率**渲染,物理尺寸约 **6–7mm**。

无需制作独立 2x 资产:使用 Godot 4 的 `TEXTURE_FILTER_NEAREST` 保证整数倍放大不模糊;缩放设置由技术美术在引擎层统一处理(`Window.content_scale_mode = integer`)。

**可读性最低要求**:Section 3.1 的剪影测试必须在 6mm 物理尺寸下通过。如通不过须回退剪影设计,不得靠「高端机上看得清」规避。

## 6. Environment Design Language

### 6.1 建筑/塔风格

**定位:地牢-城堡混合,「被废弃的人工建筑」**

- 非自然洞窟——墙体以石砖矩形(4×4px 或 8×4px 规则排列)为基本单元;锯齿边缘是石材切割痕迹
- 人工但老旧——砖缝可见(D3 灰色 1px 横纵线),局部有破损(砖角缺角 1–2px)
- 城堡元素——门用双柱宽门框(4px 边柱),楼梯是格内向上消失的 V 形阶梯暗示

| 建筑元素 | 像素表达规则 |
|---|---|
| 墙体主色 | D2 深冷灰紫为基底,4×4 砖格内 D3 点线做砖缝 |
| 墙体破损感 | 砖格角落随机缺 1px(~50% 概率);缺角用 D1 填充——「孔洞」感 |
| 地面 | D4 深灰绿棕大块底色 + D5 细小随机点;每 4×4 格一个细节点为上限 |
| 壁柱 | 每 4–6 格墙面出现一格 D3 略亮竖条——建筑结构感 |
| 门框 | 门洞两侧各 2px 宽 D3 高亮竖线;顶部 45° 斜切 2px(Section 3.2 已定义) |

**风格禁止**:有机洞穴感(不规则曲线)、现代建筑感(平滑墙面)、日式遗迹感(屋顶瓦片)。

---

### 6.2 纹理哲学——FC 像素的「点到为止」原则

| 区域 | 允许细节密度(变化像素/16×16 格) |
|---|---|
| 地面格 | ≤4px 变化 |
| 墙体格 | 8–12px 变化 |
| 道具/角色格 | 不限(它们就是内容) |
| 门/楼梯格 | 6–8px 变化(比地面高,比墙体低) |

**三步测试(每块新纹理必须通过)**:
1. **遮盖测试**:将道具/角色 sprite 叠在纹理上,确认纹理没有干扰道具识别
2. **眼动测试**:看地图 3 秒,第一注视点若落在纹理细节而非玩家/道具,纹理过强
3. **灰度测试**:纹理转灰度,确认明度不超出 D4–D5 范围——出界则重处理

最常见错误:每格都画满 12px 变化,背景比前景「更忙」。宁可地面格简到「4 个点」。

---

### 6.3 道具摆放视觉语言

道具在地面上需同时满足:**「我是个东西」+ 「我是哪类东西」**。

**三层识别机制(必须全部通过)**:
1. **亮度差**:道具明度比所在地面格高至少 2 个明度阶(M1–M9 vs D4–D5)
2. **1px 描边**:道具外轮廓统一加 D1 深炭灰 1px 描边;地面格无描边 → 有描边=有物体
3. **几何形状语义**:钥匙=齿形;药水=圆瓶;装备=剑/盾形;碎片=多边形碎块

**防混淆硬规则**:
- 道具永远不使用 D1–D5 暗色段作主色
- 深色走廊中钥匙强制使用 M1 黄金/M3 天蓝(调色板内明度最高的中间色)
- 道具与地面格不共享任何相同颜色;有交集时重新挑选道具色

---

### 6.4 楼层视觉差异化(调色板封印内)

**核心策略:换纹理图案,不换色槽**

| 变量 | 第 1–5 层 | 第 6–15 层 | 第 16+ 层 |
|---|---|---|---|
| 砖缝图案 | 水平砖缝(错缝砌法) | 方形格缝(横纵均匀) | 大块斜切缝(45°碎裂感) |
| 地面细节密度 | 极低(8×8 格 1 个点) | 低(4×4 格 1 个点) | 中(4×4 格 2 个点,不规则) |
| 墙角装饰 | 无(纯石砖) | 1px 苔藓点(D3/D4 偏绿混合) | 缺角+裂缝延伸(破败感) |
| 特殊地形密度 | 极少 | 偶发装饰柱 | 多柱子/对称装饰图案 |

**楼层进度感**:1–5 层=「塔外层」规整;6–15 层=「塔中层」岁月痕迹;16+ 层=「塔核」破碎对称,有「某人留下了什么」的氛围。

每层首帧新纹理元素闪烁 2 帧——「初见提示」,玩家无需对比上层,新元素自己「报告」它是新的。

---

### 6.5 环境叙事

**塔的背景设定**:一个已失落法术体系的实验场——某个消失的文明用它考验和培育强者。建造者不在了,塔还在运转。无居住痕迹,无家具;怪物和钥匙是「塔的机制」的一部分。

**像素叙事密度**:每层最多 1–2 个叙事细节,其余全是中性环境。

| 叙事类型 | 像素实现 | 密度控制 |
|---|---|---|
| 「有人来过」 | 墙上偶发 2×2px 符文图案(D3 轻微亮于墙体) | 每 3–5 层出现 1 次 |
| 「曾有战斗」 | 地面偶发 1×1px 暗红点(M5 稀疏点缀) | 每层 ≤3 个点,绝不成片 |
| 「塔在运转」 | 特定墙格 2 帧缓慢闪烁发光符文(M3 天蓝低频) | 每 5 层出现 1 次,仅装饰 |

**不铺满的原因**:FC 游戏的叙事力量来自玩家自己填充空白——让玩家在简单石砖里「感觉有故事」,比画出所有故事更有力量。

## 7. UI / HUD Visual Direction

> **UX 对齐状态**:美术总监草案经 UX 设计师审查。P0 冲突(触控预览)已解决;
> 3 项 UX 建议已合并。无未解决冲突。

### 7.1 HUD 布局

**位置:顶部固定栏,高度 ≤88px(16px 整数倍硬约束)**

理由:竖屏视觉重心偏上,玩家先扫顶再看格子(Z 形视线路径);底部是手势操作区,不放信息层。

**属性排布(单行,393px 宽)**

| 属性槽 | 图标 | 数字位 | 槽宽 |
|---|---|---|---|
| HP | 16px | 最多 4 位 64px | 66px |
| ATK | 16px | 最多 3 位 48px | 60px |
| DEF | 16px | 最多 3 位 48px | 60px |
| 黄钥匙 | 16px | 最多 2 位 32px | 52px |
| 蓝钥匙 | 16px | 最多 2 位 32px | 52px |
| 碎片 | 16px | 最多 3 位 48px | 59px |

合计 349px,余 28px 缓冲。碎片超出显示「999+」。

**⬆️ UX 决策 — 属性逐步解锁**:初始只显示 HP/ATK/DEF/黄钥匙(4 槽)。蓝钥匙槽在第一次遇到蓝门时弹入;碎片槽在第一次拾取碎片后弹入。降低初始认知负载。

HUD 格对齐硬规则:HUD 高度必须是 16px 整数倍,保证与游戏格子区边界精确对齐(Principle A)。

---

### 7.2 HUD 风格:Screen-space 独立面板

明确拒绝 diegetic 方式——多属性同时展示、16×16 格内头顶空间有限、回合制决策需稳定的「仪表盘锚点」。独立 HUD 面板是经典魔塔验证 30 年的模式。

---

### 7.3 字体方向

**双字体分层**:

| 层 | 用途 | 规格 |
|---|---|---|
| 数字层 | HP/ATK/DEF 数值、钥匙数、楼层数、伤害预演数字 | 等宽像素字体,8×8px 字模,**2× 整数倍渲染输出(视觉 16px)** |
| 汉字/文字层 | 怪物名、道具提示、弹窗、菜单 | 12–14px 像素风汉字字体 |

**硬规则**:
- 所有字体必须以**整数倍缩放**渲染(1×、2×、3×);禁止小数倍缩放(像素字体非整数倍会模糊)
- 汉字渲染尺寸 ≥12px,必须带 1px 暗描边
- 数字渲染尺寸 ≥14px(建议 16px)

数字与汉字不统一,但视觉调和:两套均为等宽像素风,基线对齐;混排时数字更小,形成「标签+数值」的自然层级。

---

### 7.4 图标设计风格

**尺寸:8×8px 双色像素图标(主色+高光)**

理由:HUD 是信息区,不是展示区;图标职能是「快速识别语义标签」。每多用 8px 做图标就少 8px 给数字——数字是第一等公民。

**风格:FC 方块近似图形,填充块式,统一左上角光源,1px 轮廓描边**

| 属性 | 图标形状 | 颜色 |
|---|---|---|
| HP | 5×4px 心形近似 | M8 绿+B1 白高光 |
| ATK | L形斜切(3px 柄+5px 刃向右上 45°) | M9 橙红+B3 亮橙高光 |
| DEF | 倒五边形(矩形+下方 1px 尖) | M10 冷灰蓝+B5 亮冰蓝高光 |
| 黄钥匙 | 圆头(2×2 方块)+齿形柄 | M1 黄金+M2 亮黄高光 |
| 蓝钥匙 | 同黄钥匙形状 | M3 天蓝+M4 冰蓝高光 |

**一致性规则**:所有图标高光像素统一在左上角。违反光源方向的图标视为错误素材。

---

### 7.5 触控交互:伤害预演(两步确认)

> 经 UX 审查修订:原「悬停 200ms 触发」在手机触控上不成立(手指遮挡+攻击延迟)。

**两步确认流程**:
1. **单击怪物格** → 显示预演面板(浮层在怪物格正上方或 HUD 区域,含「进攻」+「取消」按钮)
2. **点击「进攻」** → 执行攻击;**点击「取消」或空白区** → 关闭预览

预演面板视觉规格:
```
位置:怪物格正上方,或如怪物在顶部时显示在格子下方
尺寸:宽度适配内容,高度 2 行各 16px = 32px
背景:上半区 B6+B3 亮橙左侧条 = 玩家方;下半区 B6+M5 红左侧条 = 怪物方(UX 建议:用背景区分双方,不靠图标)
内容:每击 [+X 伤害] / 受击 [-Y HP]
死亡预警:整行背景改 B4 亮红 + 0.5Hz 边框闪烁(UX 建议:色盲安全双信道)
触控热区:「进攻」按钮 ≥44×44px
```

---

### 7.6 HUD 动画感

**数字跳涨**:整行高亮 + 数字弹跳 2px,峰值 128ms,消退 200ms

```
属性数字增加:
  0帧:整行背景 B6 → M1 黄金(1帧)
  1–2帧:数字上移 2px(弹起)
  3–4帧:数字回原位
  5–8帧:背景 M1 → B6(渐褪)
  总时长:~128ms

HP 减少(受伤):
  0帧:整行背景 B6 → B4 亮红(1帧)
  1–2帧:数字下移 1px(压制感)
  3–8帧:背景褪回 B6
  总时长:~128ms
```

**DEF 首次战斗微教学(UX 建议)**:第一次战斗结算后,高亮 DEF 数字旁显示一行「防御减少了 X 点伤害」,仅出现一次,永不重复。解释 DEF 的实际意义,不打断流程。

### 7.7 UI 视觉构件系统

所有 UI 面板遵循「统一构件系统」:

**面板背景**:B6 深蓝黑 + B7 冷中灰蓝 1px 边框,全直角,上下 4px/左右 6px 内边距,无投影/渐变/模糊。

**可交互按钮状态**:

| 状态 | 视觉 |
|---|---|
| 默认 | B6 背景+B7 边框+四角 1px 缺角+B1 文字 |
| 悬停/选中 | B6 背景+M1 黄金边框+B2 亮黄文字 |
| 按下 | 整体右下偏移 1px +B7 边框 |
| 禁用 | B6 背景+D3 边框+D3 灰文字 |

**三类 UI 的构件组合**:

| UI 类型 | 构件组合 |
|---|---|
| HUD 状态栏 | 面板背景(无上边框)+图标行+数字 |
| 对话/信息弹窗 | 面板背景+D1 遮罩底层+标题分割线+文字+确认按钮 |
| 菜单界面 | 面板背景+标题+列表行(16px 倍数高)+导航按钮 |

## 8. Asset Standards

> **适用版本**: Godot 4.5.2 / Compatibility 后端 / 抖音小游戏 WASM 导出（2026-06-30 re-pinned from 4.6.3）
> 标注「需实测」的条目必须在首次导出验证时逐项确认。

### 8.1 纹理规格硬约束

| 用途 | 最大尺寸 | 备注 |
|---|---|---|
| 角色/怪物 Sprite Sheet | **512×512** | 16×16 格×32帧可完整放入 |
| 地形 Tile Atlas | **512×512** | 32×32 tile 排列,共 256 tile |
| UI Atlas | **512×256** | 按 16px 格对齐 |
| 特效 Atlas | **256×256** | 帧数少,尺寸小 |
| 单张纹理绝对上限 | **1024×1024** | 超限在低端 Android WebGL 2.0 有支持风险 |

**纹理格式**:Import → Compress → Mode = **`Lossless`(PNG 无损)**。理由:像素风格 32 色硬边界,ETC2/PVRTC 有损块压缩会污染 1px 像素边。

**Power-of-2**:所有 Atlas 必须使用 POT 尺寸(256/512/1024)。WebGL 2.0 技术上支持 NPOT,但低端驱动对 NPOT 的 Mipmap 支持有缺陷,强制 POT 规避风险。「需实测」

**防模糊导入设置(每张纹理必须)**:

| 设置项 | 值 |
|---|---|
| Filter | `Nearest` |
| Mipmaps > Generate | `Off` |
| Repeat | `Disabled`(地形 Tile 按需启用) |

**项目全局默认**:`project.godot` 中设置 `textures/canvas_textures/default_texture_filter=0`。「需实测:4.6.3 中键名是否变更」

---

### 8.2 Atlas / Sprite Sheet 策略

同一 Atlas 内的多个 Sprite2D 在 Godot 4.x 2D 渲染器中自动合批为单次 Draw Call——Atlas 是降低 Draw Call 的核心手段。

**Atlas 分类方案(最多 6 张)**:

| Atlas 文件 | 内容 |
|---|---|
| `atlas_characters.png` | 玩家 + 全部怪物帧 |
| `atlas_terrain.png` | 地板、墙、门、楼梯 Tile |
| `atlas_ui.png` | HUD 图标、按钮、边框、像素数字字体 |
| `atlas_vfx.png` | 打击闪光、升级特效、拾取光晕帧 |
| `atlas_items.png` | 钥匙、药水、装备图标 |

超出 6 张时合并或改为按楼层懒加载(进入楼层时加载,离开时卸载)。

**包体预算**:全部 Atlas PNG(Lossless 压缩后)估算约 800KB–1.2MB。Godot 4.x WASM 引擎基础体积约 25–35MB(需实测)。素材包目标上限 10MB,总计 ≤50MB。

---

### 8.3 文件命名与目录约定

**命名规则**:遵循项目 snake_case 规范

| 类型 | 模式 | 示例 |
|---|---|---|
| Atlas PNG | `atlas_[category].png` | `atlas_characters.png` |
| 单体 Sprite | `spr_[entity]_[state].png` | `spr_hero_idle.png` |
| 音效 | `sfx_[action]_[variant].ogg` | `sfx_attack_hit_01.ogg` |
| BGM | `bgm_[location].ogg` | `bgm_tower_floor_01.ogg` |
| 字体 | `fnt_[name]_[size].png` | `fnt_pixel_8.png` |

**禁止**:文件名含空格、中文、大写字母、特殊符号(`.`、`-`、`_` 除外)。

**assets/ 目录结构**:
```
assets/
├── art/
│   ├── atlases/        # 所有合并 Atlas PNG + .import
│   ├── sprites/        # 独立 Sprite(仅不适合入 Atlas 的例外)
│   ├── fonts/          # 位图字体 PNG + .import
│   └── palette/        # 调色板参考文件(.gpl/.png,不导入引擎)
├── audio/
│   ├── sfx/            # 音效 .ogg
│   └── bgm/            # 背景音乐 .ogg
├── shaders/            # .gdshader(由 godot-shader-specialist 管理)
├── vfx/                # ParticleProcessMaterial .tres
└── data/               # 游戏数据 JSON/CSV(怪物表、楼层配置等)
```

### 8.4 动画帧规格

**节点选择**:
- `AnimatedSprite2D` → 角色/怪物待机/攻击/受击(简单帧序列,低开销)
- `AnimationPlayer` → 升级特效、UI 弹出、需要同时动画化 position/信号回调的复合动画

**帧数约定**:

| 状态 | 帧数 | 备注 |
|---|---|---|
| 待机(idle) | 2–4 帧 | 呼吸/闪烁感即可 |
| 攻击(attack) | 3–4 帧 | 预备→出手→收手 |
| 受击(hurt) | 2 帧 | 闪白→恢复 |
| 死亡(die) | 3–4 帧 | 倒下/消散 |
| 移动 | 2–4 帧 | 可选;网格步进可仅用位移 tween |
| 特效(闪光/升级) | 4–6 帧 | vfx atlas 帧 |

单个实体全状态帧总数 ≤ **16 帧**(保证单角色在 512×512 Atlas 内可容纳 8–10 个不同实体)。

**帧时长建议(目标 30fps,战斗节奏 0.3s/回合)**:

| 动画 | 总时长 | 帧数 | 推荐 fps 设置 |
|---|---|---|---|
| 攻击 | 0.18s | 3 帧 | fps=16 |
| 受击 | 0.12s | 2 帧 | fps=16 |
| 待机 | 0.4–0.8s 循环 | 4 帧 | fps=8 |
| 升级特效 | 0.3s | 6 帧 | fps=20 |
| 死亡 | 0.4s | 4 帧 | fps=10 |

---

### 8.5 音频素材规格

**格式:OGG Vorbis 作为所有音频的标准格式**。WAV 仅限 <0.1s 需零延迟触发的极短音效;不使用 MP3(WASM 解码库体积较大)。「需实测:OGG 在抖音 WebView WebAudio 下的解码延迟」

| 参数 | 值 |
|---|---|
| 采样率 | 44100 Hz |
| 声道 | 音效 Mono,BGM Stereo |
| 码率 | 音效 64–96 kbps,BGM 96–128 kbps |
| 音效单文件上限 | 100KB |
| BGM 单文件上限 | 500KB |
| 全部音频总上限 | **5MB** |

---

### 8.6 Draw Call 优化

**地形强制使用 `TileMapLayer`**(Godot 4.3+ 替代废弃的 `TileMap`)。整层地图约 1–2 次 Draw Call vs 手工拼 Sprite 的 N 次。

**Draw Call 分配预算参考**:

| 来源 | 估算 |
|---|---|
| 地形 TileMapLayer | 1–2 次 |
| 角色+怪物 Sprite2D(同 Atlas 合批) | 1–3 次 |
| UI CanvasLayer(Atlas 合批) | 2–4 次 |
| 特效 Atlas | 1–2 次 |
| 合计 | ~5–11 次(远低于 ≤50 上限) |

**TileMapLayer 已知问题(需实测 4.6.3)**:
- Y-Sort 与 TileMapLayer 混用时排序可能错误 → 怪物/玩家单独放 Node2D 层,地形层不启用 Y-Sort
- Compatibility 后端对 TileMapLayer 自定义着色器有限制 → 不对地形 Tile 使用自定义材质

### 参考 1 — 勇者斗恶龙 FC 版(Dragon Quest I/II, NES)

**取用**:地下城「深色高密度墙体 + 极低细节地面格」的两极对比策略。墙砖是显著的 4×4 像素块状纹理;地面几乎是纯色空白。玩家的可移动区域因此「一眼读清楚」,无需任何半透明或高光提示。这就是 Section 6.2「地面 ≤4px、墙体 8–12px」密度分层规则的来源。

**不取用**:「无地图信息」的随机遭遇设计。本作把怪物预先陈列在地图上,视觉语言完全不同;仅取墙/地面密度对比关系。

---

### 参考 2 — 魔塔原型系列(PC-88 1983 / Win95 中国版)

**取用**:「黄钥匙/黄门、蓝钥匙/蓝门」的颜色-功能强绑定约定。这套语义在中国玩家群体中形成 30 年以上的条件反射式认知。Section 4.2 的 M1 黄金/M3 天蓝语义色直接继承这套体系——利用已有玩家认知降低学习成本,而非「致敬」。

**不取用**:任何具体像素图形风格。原版魔塔的像素美学标准与本作无关;取的是「色彩语义的约定」。

---

### 参考 3 — Downwell(Ojiro Fumoto, 2015)

**取用**:「克制背景放大反馈对比度」的杠杆原理。Downwell 以约 3 色极端克制运行,爆炸/连击特效是全屏饱和度最高的时刻。玩家视觉系统被校准到「稍微亮一点就是重要事件」的敏感状态——背景越克制,反馈的峰值不需要太高就能产生震撼感。

**这直接服务于 Principle C**:评估「成长反馈是否够亮」时,标准不是绝对亮度值,而是「相对于本作克制背景的对比度差」。Downwell 提供了可操作的校准方法论。

**不取用**:极简到几乎无颜色的整体风格。本作保留 32 色丰富语义系统;只借鉴「对比度杠杆原理」。

### 参考 4 — 像素艺术家 Fool(ドット絵作家 Fool, @Fool_dq)

**取用**:16×16 格内「主色 + 暗色 + 高光」三色分层技法——每个身体区块有清晰的受光面和阴影面,不超过 3–4 个色阶,在 32 色约束内制造可读性极高的三维感知。**高光统一在左上角**,与 Section 7.4「所有图标高光统一在左上角」完全对应——让整套资产在风格上产生内在一致性,玩家感受到「这些角色存在于同一个光照世界里」。

**不取用**:任何具体图形形状或配色。取用的仅是「三色分层 + 统一光源方向」这一结构性技法,作为美术绘制角色时的操作规范。

### 参考 5 — Spelunky Classic(Derek Yu, 2008)

**取用**:「功能形状优先于装饰细节」的 Tile 设计策略。每块 Tile 首先用最少像素数表达「这是墙/地面/可交互点」,然后才在不破坏可读性前提下加装饰。具体技法:墙体 Tile 用「外轮廓描边 + 中央简单重复图案」双层结构;地面用单色大块 + 稀疏随机点。这与 Section 6.1 的砖缝规则高度对应——可作为新楼层地形 Tile 设计时的**可读性校准标准**直接参照。

**不取用**:有机洞穴/随机地形的视觉风格。Spelunky 的空间是随机生成+有机洞窟,有大量曲线和倾斜地面。本作是人工建筑全直角,空间组织方式完全不同。取的只是「Tile 设计优先顺序」的操作哲学。
